# 4.1 ANALISA KEBUTUHAN SISTEM

Analisa kebutuhan sistem merupakan tahapan untuk mengidentifikasi dan mendokumentasikan apa yang harus dilakukan oleh sistem agar dapat memenuhi kebutuhan pengguna. Kebutuhan sistem terbagi menjadi dua kategori utama, yaitu kebutuhan fungsional yang menggambarkan proses atau fungsi yang harus dilakukan sistem, dan kebutuhan non-fungsional yang mendefinisikan properti dan batasan dari sistem.

## 4.1.1 Kebutuhan Fungsional

Kebutuhan fungsional adalah fungsi-fungsi yang harus dimiliki oleh sistem untuk memenuhi kebutuhan pengguna. Pemodelan fungsional sistem menggambarkan proses atau fungsi yang harus dilakukan sistem untuk memenuhi persyaratan. Berdasarkan hasil analisis, Sistem POS DW memiliki tiga aktor yang berinteraksi dengan sistem, yaitu Admin, Kasir, dan Manager. Adapun kebutuhan fungsional Sistem POS DW dapat dilihat pada tabel 4.1 sebagai berikut:

## 4.1.2 Kebutuhan Non-Fungsional

Kebutuhan non-fungsional sistem mendefinisikan properti dan batasan dari sistem. Kebutuhan non-fungsional ini bisa lebih krusial dibandingkan kebutuhan fungsional, karena jika tidak terpenuhi, sistem mungkin tidak bisa digunakan. Berdasarkan kebutuhan fungsional sistem yang telah dijelaskan sebelumnya, diharapkan sistem yang dirancang dapat memiliki hal-hal sebagai berikut:

**Tabel 4.2 Kebutuhan Non-Fungsional Sistem POS DW**

| Kode | Aspek | Kebutuhan | Deskripsi |
|------|-------|-----------|-----------|
| RNF-01 | **Performance** | Response Time | Sistem harus merespons request user dalam waktu maksimal 2 detik untuk operasi normal |
| RNF-02 | **Performance** | Page Load Time | Halaman harus dimuat dalam waktu maksimal 3 detik pada koneksi internet standar |
| RNF-03 | **Performance** | Concurrent Users | Sistem harus dapat menangani minimal 50 user yang aktif bersamaan |
| RNF-04 | **Performance** | Transaction Processing | Sistem harus dapat memproses minimal 100 transaksi per jam |
| RNF-05 | **Security** | Password Encryption | Password user harus dienkripsi menggunakan algoritma bcrypt |
| RNF-06 | **Security** | HTTPS | Sistem harus menggunakan protokol HTTPS untuk semua komunikasi |
| RNF-07 | **Security** | SQL Injection Prevention | Sistem harus terproteksi dari serangan SQL Injection menggunakan prepared statements |
| RNF-08 | **Security** | XSS Prevention | Sistem harus terproteksi dari serangan Cross-Site Scripting (XSS) |
| RNF-09 | **Security** | CSRF Protection | Sistem harus terproteksi dari serangan Cross-Site Request Forgery (CSRF) |
| RNF-10 | **Security** | Session Security | Session harus memiliki timeout otomatis setelah 120 menit tidak aktif |
| RNF-11 | **Security** | Role-Based Access Control | Sistem harus menerapkan kontrol akses berbasis role (RBAC) |
| RNF-12 | **Usability** | User Interface | Interface harus intuitif dan mudah digunakan tanpa training khusus |
| RNF-13 | **Usability** | Responsive Design | Interface harus responsive dan dapat diakses dari desktop, tablet, dan mobile |
| RNF-14 | **Usability** | Error Messages | Sistem harus menampilkan pesan error yang jelas dan informatif |
| RNF-15 | **Usability** | Loading Indicators | Sistem harus menampilkan loading indicator saat memproses request |
| RNF-16 | **Compatibility** | Browser Support | Sistem harus kompatibel dengan browser modern: Chrome, Firefox, Edge, Safari (versi terbaru) |
| RNF-17 | **Compatibility** | Screen Resolution | Sistem harus dapat ditampilkan dengan baik pada resolusi minimal 1366x768 |
| RNF-18 | **Reliability** | Uptime | Sistem harus memiliki uptime minimal 99% (downtime maksimal 7.2 jam per bulan) |
| RNF-19 | **Reliability** | Data Backup | Sistem harus memiliki mekanisme backup database otomatis setiap hari |
| RNF-20 | **Reliability** | Error Recovery | Sistem harus dapat recovery dari error tanpa kehilangan data transaksi |
| RNF-21 | **Maintainability** | Code Quality | Source code harus mengikuti PSR-12 coding standard |
| RNF-22 | **Maintainability** | Documentation | Sistem harus memiliki dokumentasi kode dan API yang lengkap |
| RNF-23 | **Maintainability** | Modular Architecture | Sistem harus menggunakan arsitektur modular untuk memudahkan maintenance |
| RNF-24 | **Scalability** | Database Scalability | Database harus dapat menampung minimal 1 juta record transaksi |
| RNF-25 | **Scalability** | Horizontal Scaling | Sistem harus mendukung horizontal scaling untuk menangani traffic tinggi |
| RNF-26 | **Localization** | Timezone | Sistem harus mendukung pengaturan timezone (default: Asia/Jakarta) |
| RNF-27 | **Localization** | Currency Format | Sistem harus mendukung format mata uang Rupiah (Rp) |
| RNF-28 | **Localization** | Date Format | Sistem harus menggunakan format tanggal Indonesia (DD/MM/YYYY) |
| RNF-29 | **Availability** | 24/7 Access | Sistem harus dapat diakses 24 jam sehari, 7 hari seminggu |
| RNF-30 | **Compliance** | Data Privacy | Sistem harus mematuhi regulasi perlindungan data pribadi |

*Sumber: Hasil Analisis Kebutuhan (2026)*

## Ringkasan

Berdasarkan analisa kebutuhan sistem yang telah dilakukan, Sistem POS DW dirancang dengan mempertimbangkan:

1. **35 Kebutuhan Fungsional** yang mencakup seluruh fitur utama sistem dari autentikasi hingga laporan, yang terbagi dalam lima modul yaitu Autentikasi, Manajemen Master Data, Transaksi Penjualan, Laporan dan Analitik, serta Pengaturan Sistem.
2. **30 Kebutuhan Non-Fungsional** yang menjamin kualitas sistem dalam aspek performance, security, usability, compatibility, reliability, maintainability, scalability, localization, availability, dan compliance.
3. **Spesifikasi Hardware dan Software** yang sesuai untuk development dan production environment dengan stack teknologi modern berbasis Laravel 13, Livewire 4, dan MySQL 8.

Seluruh kebutuhan yang telah didefinisikan akan menjadi acuan dalam tahap perancangan dan implementasi sistem yang dijelaskan pada sub-bab selanjutnya.
